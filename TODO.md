## 🔴 Критично (ломает генерацию)

### 1. Починить retry-цикл → thinking mode
**Проблема:** любой retry запускает Gemma-4 в режим размышлений (145–430 сек).
Два независимых источника retry:

#### 1a. Баг: source_fact_mismatch демотируется и сразу добавляется обратно
- `llm/client.py`: код делает `error_categories.pop("source_fact_mismatch")`, затем
  сразу вызывает `_apply_source_fact_consistency_checks()` — она снова добавляет те же ошибки
- **Фикс:** вызывать `_apply_source_fact_consistency_checks()` ДО демотирования,
  или убрать вызов вовсе (heuristic check — не нужен при хорошем промпте)

#### 1b. "до 130" / один HR-порог → hr_low: null → validator error → retry
- Пользователь пишет "заминка 2 км (до 130 уд/мин)" — только верхний порог
- Модель: `hr_low: null` или `hr_low: 130, hr_high: 130`
- Валидатор: `hr_low must be less than hr_high` → retry

  **Варианты фикса (выбрать один):**
  - A. `MAX_RETRIES = 0` — принимать первый ответ LLM всегда (быстро, грубо)
  - B. В YAML repair-слое: если cooldown-шаг `hr_low >= hr_high` → заменить на `dist_open`
  - C. В промпте: добавить правило «если задан только верхний порог HR, используй dist_open»
  - D. Препроцессинг источника: "до X" → "115–X" (подставить разумный нижний порог)

  _Рекомендуется: сначала C (одна строка в промпте), потом B как fallback_

## 🟡 Важно (UX / надёжность)

### 2. Кнопки доставки пропадают при рестарте бота
При перезапуске бота старые `InlineKeyboardMarkup` в чате перестают работать
(callback handler жив, но `UserState` сброшен → `pending_zip_path = None`).
Нужно: при `delivery:zip` / `delivery:garmin` проверять наличие файла,
если нет — просить пользователя пересобрать (`/build`).

### 3. /cancel во время generating не работает
Флаг `cancel_requested` проверяется в `_execute_build_job`, но не в `_process_plan` (LLM-генерация).
LLM-запрос на 5+ минут нельзя прервать после отправки.
Варианты: timeout на запрос, или хотя бы предупреждение «отмена будет после генерации».

### 4. ZIP не удаляется если пользователь игнорирует кнопки
`pending_zip_path` остаётся на диске если пользователь отправил новый план без выбора доставки.
Нужно: в `reset_state()` удалять файл если он ещё существует (уже частично реализовано — проверить).
